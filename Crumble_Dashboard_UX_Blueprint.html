Section 01

Overall UI/UX Strategy & Philosophy

A dashboard ecosystem designed as two distinct but unified experiences: an executive intelligence platform for leadership decision-making, and a real-time operational control layer for field teams.

1.1 — Core UX Philosophy

The Crumble dashboard ecosystem is not a reporting tool. It is a decision-support operating system — a product that different users live inside for different purposes. The design philosophy rests on five pillars:

Clarity Over Decoration

Every pixel earns its place. No chart exists without a decision it informs. No metric appears without context (trend, comparison, threshold).

Actionability Over Information

Operational screens must answer "What do I do next?" Executive screens must answer "What changed and why?"

Layered Complexity

Surface layer is scannable in 3 seconds. First click reveals trends and comparisons. Second click reaches root-cause detail. Depth is available; it is never forced.

Exception-First Design

When everything is normal, the screen confirms it calmly. When something breaks threshold, the UI makes it impossible to miss — through color, position, motion, or persistent alert.

Context-Rich Metrics

No number exists alone. Every KPI shows: current value, trend direction, variance from target/prior period, and severity classification.

Unified but Differentiated

Executive and operational layers share one design system (typography, color, components) but diverge in density, refresh cadence, interaction model, and visual emphasis.

1.3 — Dashboard Usage Model

Morning Cadence (Executives): CEO opens Executive Command Center → scans health score → reviews overnight anomalies → drills into Marketing ROI or Outlet Performance → closes within 5–8 minutes with full situational awareness.
Shift Cadence (Operations): Outlet Manager opens Live Operations Board → checks pending orders and queue depth → reviews Bake Planner for next 2 hours → acknowledges inventory alerts → glances at waste tracker → repeats every 30–60 minutes throughout shift.
Campaign Review (Marketing): CMO opens Marketing ROI Dashboard → filters to current LTO campaign → compares platform performance → reviews creative fatigue indicators → checks geo response → exports report for leadership meeting.

1.4 — LTO-Centric Design Integration

Crumble's monthly limited-time offer (LTO) product launch is a business-defining event that ripples across every department. The UX must treat the LTO cycle as a first-class temporal dimension, not an afterthought:

  • Every dashboard must support filtering by "Current LTO" / "Previous LTOs" / "Core Menu Only"
  • Launch timelines appear as visual overlays on trend charts (vertical markers showing launch date)
  • Pre/Post comparison logic is built into every metric: "performance before launch vs during launch vs after launch"
  • LTO Health Card appears on the Executive Command Center showing days since launch, demand curve, city response, and go/no-go signal for permanence
  • Production and inventory dashboards show LTO-specific stock, thaw schedules, and waste risk

5.2 — Filter Bar Behavior

  • Sticky positioning: Filter bar sticks below the top bar on scroll. Content scrolls beneath it.
  • Collapse on mobile: Collapses to a "Filters (3)" chip that expands to a bottom sheet on tap.
  • Active filter indicators: Applied filters show as colored chips. "Clear All" button appears when any non-default filter is active.
  • Saved Views: Users can save current filter + page combinations as named views (e.g., "My Islamabad Weekly Review"). Accessible from user menu or filter bar.
  • Role Defaults: On login, filters auto-apply based on role scope. Outlet Manager: locked to outlet. Regional Manager: locked to assigned cities.
  • URL State: All filters are reflected in URL parameters, enabling bookmarking and sharing of specific views.
  • Reset: "Reset to Defaults" restores role-based default filter state.

6.5 — Density Modes

Comfortable (Executive Default)

Larger cards (min-height 120px), 24px card gaps, 16px inner padding, 28px KPI values. Optimized for scannability and premium feel. Used on Executive Command Center, Marketing ROI, Customer/Loyalty.

Compact (Operations Default)

Smaller cards (min-height 80px), 12px card gaps, 12px inner padding, 20px KPI values. Optimized for maximum information density. Used on Live Ops Board, Bake Planner, Inventory, Data Quality.

6.6 — Chart Styling Rules

  • Line charts: 2px stroke, dot markers on hover only, area fill at 5% opacity. Grid lines at 1px #e2e8f0.
  • Bar charts: 8px radius on top corners, 4px gap between bars. Highlight bar on hover with tooltip.
  • Donut charts: Use only when showing composition of 3–5 segments. Inner label shows total. Legend below chart.
  • Heatmaps: 5-step sequential palette (lightest to darkest of one hue). Used for daypart analysis, geo performance.
  • Sparklines: 1.5px stroke, no axes, no grid. In-line within KPI cards or table cells. Green/red coloring based on trend direction.
  • Tables: Alternating row shading off by default. Hover highlight on. Sortable columns with arrow indicators. Pinned headers on scroll.
  • LTO Overlays: Vertical dashed line with label on time-series charts marking LTO launch dates. Pink/magenta color to distinguish from standard annotations.

6.7 — Iconography

Use a single icon library consistently throughout: Lucide Icons (open-source, consistent stroke weight, scales well at small sizes). Supplement with custom status icons for operational states only where Lucide lacks coverage.

  • Navigation icons: 20px, 1.5px stroke
  • Inline icons: 16px, 1.5px stroke
  • Status icons: 14px, filled variants for critical states
  • No decorative icons — every icon must convey meaning

14 Accessibility & Usability Standards

Accessibility is not a compliance checkbox. In a data-dense environment like Crumble's dashboard ecosystem, accessibility principles directly improve usability for every user — from the CEO scanning KPIs on a laptop in a bright meeting room to the outlet manager checking the wallboard from across the kitchen.

14.1 — Color & Contrast

14.2 — Keyboard Navigation

14.3 — Screen Reader Support

  • ARIA landmarks: navigation, main, complementary, banner on all pages.
  • Live regions: Alert banners use aria-live="assertive". Data refresh uses aria-live="polite".
  • Chart descriptions: Every chart has an aria-label describing the chart type and key takeaway. Screen readers announce: "Bar chart showing monthly revenue trend. Current month: 12.4 million, up 8% from last month."
  • Table announcements: Column headers include sort state. Row count announced on load.
  • KPI cards: Read as: "Revenue: 12.4 million. Up 8 percent from last period. Target: 13 million."

14.4 — Dense-Data Readability

When screens contain large amounts of data (tables with 50+ rows, dashboards with 12+ components), these rules prevent cognitive overload:

  • Visual hierarchy: Maximum 3 levels of visual weight per page (primary KPIs → secondary charts → supporting tables).
  • White space: Minimum 16px gap between cards. 24px between sections. Never less than 8px between any two elements.
  • Row striping: Alternating row backgrounds on tables with > 10 rows.
  • Sticky headers: Table column headers stick on scroll. Page section headers stick briefly.
  • Progressive disclosure: Default to showing top 10 rows. "Show More" or pagination for the rest.
  • Number formatting: Consistent: thousands separator (1,234), 1 decimal for percentages (12.4%), abbreviated millions (12.4M). No raw long numbers.
  • Alignment: Numbers right-aligned. Text left-aligned. Headers match their column alignment.

14.5 — Motion & Animation Preferences

  • Respect prefers-reduced-motion media query. When enabled: disable chart animations, transitions become instant, skeleton shimmer becomes static gray.
  • No auto-playing video or aggressive animation anywhere in the system.
  • All transitions under 300ms. No animations longer than 600ms (KPI counter roll-up is the longest).

15.4 — Page Anatomy Checklist

Every dashboard page must include these elements in this order:

  1. Page Title — H1, left-aligned, with breadcrumb above if nested deeper than level 1.
  2. Alert Banner — Conditional: only appears when exceptions are active for this page's scope.
  3. KPI Row — 4-6 summary metrics with trend and comparison. These are the "pulse" of the page.
  4. Tab Bar — If the page has sub-views (most executive dashboards do).
  5. Visual Content — Charts, tables, maps, operational boards. Organized by information flow (overview → detail).
  6. AI Insights — Optional but recommended: 1-2 AI-generated insight cards summarizing what the data means.
  7. Footer Actions — Export, share, schedule report. Subtle, bottom-aligned.

16.4 — Iconography Style

  • Icon Library: Lucide Icons (open source, consistent 24px grid, 1.5px stroke).
  • Sizes: 16px (inline with text), 20px (buttons and nav), 24px (page-level actions), 32px (empty states).

16.6 — Premium Aesthetic Principles

  • Restraint over decoration: No gradients on data elements. No 3D effects. No drop shadows on charts. Flat is premium.
  • Whitespace is intentional: Every gap, margin, and padding is a deliberate choice, not a leftover. The system breathes.
  • Brand warmth through accent: Crumble orange (#E8500A) is used as a strategic accent — active states, primary CTAs, and the sidebar logo. It should feel like a warm punctuation mark, not a dominating color.
  • Data density without clutter: Use Compact density mode for operational dashboards. Reduce row height, pad cells tighter, but never below readable thresholds.
  • Consistency builds trust: If a user sees Revenue as a blue line on one page, it must be the same blue everywhere. The chart palette is not just aesthetic — it's a semantic language.
  • Animation as information: KPI counter roll-ups tell the user "this just updated." Chart draw animations show "new data arrived." Motion is not decoration — it's communication.

18.6 — Design Handoff Checklist

For each screen moving to development, the design handoff package must include:

  1. High-fidelity mockup — Figma file with all states (default, loading, empty, error).
  2. Responsive variants — Desktop, tablet, mobile (where applicable).
  3. Component specification — Which shared components are used, with any parameter overrides.
  4. Interaction annotations — What happens on click, hover, drag for each interactive element.
  5. Data mapping — Which API endpoint feeds each component. Expected data shape.
  6. Exception states — Alert conditions, thresholds, what happens when data is missing or stale.
  7. Accessibility notes — ARIA labels, keyboard flow, screen reader announcements.
  8. Acceptance criteria — Quantified definition of done for QA.
